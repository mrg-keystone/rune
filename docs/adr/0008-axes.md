# ADR 0008 — Axes: language vs codegen profiles (D7)

Status: **Accepted** · Closes the decision behind G13

## Context

Codegen output depends on a **target profile** — language, runtime, validator
lib, module system. The Rust engine already treats this as a swappable axis
(`CONFIG = "ts-deno-native-class-validator-esm"` selects one of many
`cli/src/configs/<profile>/` trees), but the Studio bakes these into the registry
as fixed singletons (`vars.runtime: "deno"`, `vars.validatorLib: "class-validator"`)
and ships one set of templates. "Editing the language" keeps colliding with
"switching stacks."

## Decision

Separate **target-independent `language`** (tags, grammar, lint) from **N
selectable per-target `codegen` profiles**. The artifact carries a `profiles[]`
list; one language drives many targets. Switching profiles regenerates against a
different target without touching the language.

## Consequences

- WO-3's `artifact.schema.json` has separable `language` / `lint` / `codegen`
  (per-profile) sections plus `profiles[]`.
- WO-4 resolves codegen by selected profile; WO-7 adds profile pick/clone UI.
- Gate (WO-7): switching profiles regenerates a different target with the
  language unchanged.

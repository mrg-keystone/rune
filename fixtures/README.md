# Verification fixtures

The oracle the rebuild is gated against (see `rune/new/studio/verification.md`).
Run the ladder with `deno task verify` (exit 0/1) or one gate with
`deno task verify --gate L3`. Recapture goldens **intentionally** with
`deno task verify --update-goldens` — an unreviewed golden diff is the failure
mode to guard against, so review the diff before committing.

## Layout

```
fixtures/
  corpus/valid/*.rune     specs the current engine parses cleanly (0 errors)
  corpus/invalid/*.rune   specs the current engine rejects (>=1 parse error)
  golden/parse/*.json     L2 — parse(spec) AST, captured from today's engine
  golden/manifest/*.json  L3 — planManifest(spec) file tree (paths + content)
  golden/lint/*.json      L4 — lint --json over a fixture project
  projects/<gen>/         materialised from a [MOD] corpus spec (regenerated by
                          --update-goldens); linted for L4 — stays lint-clean
  projects/dirty/         hand-authored tree with deliberate violations so L4
                          proves the linter fires (12 findings across 5 rules)
```

## Corpus coverage

Every tag (`[REQ] [MOD] [ENT] [PLY] [CSE] [NEW]/[CTR] [RET] [TYP] [DTO] [NON]`),
all six boundary prefixes (`db: fs: mq: ex: os: lg:`), faults (0/1/many),
polymorphism (single/many/nested), `:core` on `[DTO]`/`[TYP]`, static `::` vs
instance `.`, inline `{}` DTOs, multi-module specs, the realistic end-to-end
`example-e2e`, plus the parser's error paths in `invalid/`.

## Known engine gap characterised here

`valid/poly-nested.rune` contains a `[PLY]` nested inside a `[CSE]`. The spec
calls this disallowed, but **today's engine parser does not reject it** — the
`[PLY]` handler restarts a top-level poly block under the `[REQ]` and silently
drops the case nesting (`rune-parse/mod.ts:292`; the guard at `:589` is
unreachable for a real `[PLY]` line). It is therefore filed under `valid/`
(0 parse errors) and its parse golden captures the lenient behaviour. Closing
this gap belongs to WO-4c (artifact-driven parser) / WO-4d (lint), at which
point it should move to `invalid/` with a golden diff that documents the change.
